Understanding Tokens & Context Windows

  • What's a token? Roughly 4 characters or 0.75 words
  • Examples: "Hello" = 1 token, "Hello world" = 2-3 tokens
  • Context window: How much AI can "see" at once
  • Math: 100K tokens ≈ 75,000 words ≈ 300-page book
  • The problem: When you hit the limit, AI forgets earlier context
This is why fresh chats & strategic context matter. You're managing limited space.

Why Starting Fresh Is Better

  • The research: 11 of 12 AI models drop to 50% worse performance at 32k tokens
  • Context anxiety: AI "panics" when it thinks context is full and takes shortcuts
  • Lost in the middle: AI forgets what's in the center of long conversations
  • Summaries fail: Each compression loses details (like CliffsNotes: you get the main idea, but lose the important stuff)
  • The cascade: Compress once = some details lost; Compress twice = vaguer; Compress 3+ times = unreliable
The fix: Fresh chat > Compressed chat. Always.
When 100+ messages in, start new chat with a checklist instead of relying on memory

Context Engineering (LCMP)

Long-term Context Management Protocol:

1. Fresh chat per feature (no contamination)
2. Document architecture in every prompt
3. Clear tool results when switching
4. Maintain structured notes across sessions
Think of it like Git branches for your AI conversations

The Three-Step Approach: Talk → Plan → Build

Step 1: TALK with AI about your idea
↓ "What do we want to build?"
↓ Ask questions, work out confusing parts

Step 2: MAKE a checklist
↓ Write down all the things to build
↓ Break it into small, doable tasks

Step 3: BUILD one task at a time
↓ New chat for each task
↓ Check it works
↓ Move to the next one

Result: Way fewer mistakes, cleaner work
The key idea: Plan first, code second
Like an essay outline = better than writing randomly

Pattern 1: Architecture-First

"I'm working on a Go project using Echo and SQLite.
I've organized my code like this: /handlers for the
web stuff, /models for database stuff, /views for HTML.

I want to add a comment system for blog posts. Can you
look at how I structured my contact handler and user
model, then build the comment system the same way?

Here's my existing code: [paste code]"
Explain the big picture first, then ask for help

Pattern 2: Test-Driven Prompting

"Write tests that validate password strength. Make
sure you test edge cases like:
- Empty passwords
- Passwords that are too short
- Ones missing numbers or uppercase letters
- Security issues like <script> tags (XSS attempts)

Search the internet for best practices on password
validation. Then write the ValidatePassword function
to pass all those tests. Make it secure."
Tell AI what you want tested, let it write tests AND code

Pattern 3: Two-Stage Security

Stage 1: Build

"Help me build a comment form
where users can submit comments.
Just get it working - form, save
to database, display comments.
Don't worry about security yet."

Stage 2: Secure

"Now act like a security expert.
Review this comment code. Tell me
everywhere a hacker could break in.
Check for: XSS attacks, SQL
injection, CSRF. Be thorough."
Build first, then secure. Way fewer bugs this way

Pattern 4: Few-Shot Examples

"Here's how I built my contact handler:
[paste your existing code]

I like this style - the way it checks
if it's GET or POST, how it validates,
and how it returns responses.

Can you build a profile handler that
follows this exact same pattern and
structure? Keep my code style consistent."
Show AI an example, it learns your style

Common Intermediate Mistakes

  • Context contamination: Mixing unrelated tasks in same chat
  • Missing dependencies: Not showing related files
  • Vague requirements: "Make it faster" vs "Under 200ms"
  • Ignoring edge cases: Empty data, special chars, Unicode
  • Blind trust: Not reviewing for security/correctness
